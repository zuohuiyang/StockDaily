# 故障排查（TROUBLESHOOTING）

本文件用于快速定位 OpenClaw/调度在调用 StockDaily 时常见的失败原因。推荐按顺序排查，避免在“数据缺失”与“运行环境错误”之间来回切换。

## 先看这三个关键信号

- 工作目录：是否在仓库根目录运行（或是否显式传入 `--db` 为绝对路径）
- 退出码：`0` 表示成功生成日报；非 `0` 表示关键错误
- 交付物：成功后应存在 `daily_reports/YYYY-MM-DD_report.md`，且标准输出会打印该归档路径

## 常见问题与处理

### 1) `No module named stock`

原因通常是 Python 找不到 `src/` 下的包。

处理：
- 确保工作目录为仓库根目录
- 确保 `PYTHONPATH` 包含 `src/`
  - Windows PowerShell：`$env:PYTHONPATH = (Join-Path (Get-Location) 'src')`
  - Linux/macOS：`export PYTHONPATH="$(pwd)/src"`

### 2) 数据库不在预期位置/读错库

默认数据库是工作目录下的 `portfolio.db`。如果调用方在其它目录运行，可能会创建/读取到另一个空库，从而导致日报缺失数据或无法确定日期。

处理：
- 固定工作目录为仓库根目录运行
- 或显式传入 `--db` 使用绝对路径

### 3) 日报里很多 `-`，汇总看起来不对

本项目对缺失数据不做估算：缺少行情或汇率时会显示为 `-`，并且不会计入汇总，避免误导。

可能原因：
- 没有为目标日期采集到 EOD 行情或汇率
- 首次接入，没有回补历史区间
- 采集命令 `--symbols` 与日报命令 `--position` 标的不一致

处理建议：
- 对目标日期先跑一次采集：`public-backfill --start=目标日期 --end=目标日期 --symbols 同一批标的`
- 首次接入先回补一段历史区间，再生成日报
- 确保 `--symbols` 与 `--position` 使用同一批标的集合

### 4) 采集阶段网络失败/超时/限流

表现通常是采集命令退出码非 0，或某些标的缺失导致日报出现 `-`。

处理建议：
- 缩小范围复现：先用单个标的 + 单日采集
- 稳定后再扩大到整批标的
- 若生产环境网络受限，优先确认 DNS/HTTPS 出站策略与代理设置

### 5) 生成日报失败，但采集成功

常见原因：
- 未传 `--date` 且数据库中无法确定“可用的最新 EOD 日期”
- 输出目录不可写、磁盘权限问题、文件被占用

处理建议：
- 生产调用建议显式传 `--date YYYY-MM-DD`，不要依赖“自动选择最新日期”
- 确保输出目录可写（默认 `daily_reports/`）

## 复现与反馈时请提供

- 两条命令的完整调用（采集与日报），包含全部参数
- 调用时的工作目录（cwd）
- `--db` 实际指向的数据库路径
- 目标日期与持仓列表（`--position`）
- 失败时的退出码与标准输出/标准错误全文
