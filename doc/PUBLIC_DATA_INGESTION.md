# 公网数据采集层（仅收盘价）方案

本文档约定“数据采集层”中**公网数据**部分的目标、数据范围与运行方式：只采集收盘价（EOD），不做实时行情；以“初始化回填历史 + 每日增量同步 + 新增标的回填”的方式维护数据库。

## 目标

- 为日报/分析提供稳定的事实数据：A 股收盘价、美股收盘价、USD/CNY 汇率。
- 数据可追溯、可重复拉取：允许重试/覆盖式更新，但不做插值、猜测或估算。
- 每日只需要跑一次增量任务即可，产出数据库中的“最新可用交易日快照”。

## 数据范围（3 类）

- A 股价格：按交易日的收盘价（EOD close）。
- 美股价格：按交易日的收盘价（EOD close）。
- USD/CNY 汇率：按日期的汇率（建议与价格日期对齐使用）。

## 核心原则

- **只收盘价**：不采集实时价格；当天未收盘时，不写“当天”价格。
- **不做修复/估算**：缺数据就缺数据；日报应能识别缺口并跳过或提示。
- **允许补拉/重试**：数据源超时、限流、节假日等导致的缺口，后续可通过再次拉取补齐（这是“同步”，不是“修复”）。
- **允许覆盖式更新**：同一标的同一日期，如果数据源回溯修正（复权/拆股/更正），可通过 upsert 覆盖旧值。

## 运行方式

公网数据采集拆成两条子流程（都属于“采集”，不属于“治理/修复”）：

### 1) 历史回填（Backfill）

适用场景：
- 首次初始化数据库；
- 新增标的后补齐其历史价格。

输入：
- 标的列表（A 股 / 美股）；
- 起止日期（推荐起点：你的第一笔交易日或你希望开始统计的日期）。

输出：
- 写入 `stock_prices`：按“标的 + 交易日”保存收盘价；
- 写入 `exchange_rates`：保存 USD/CNY（按日期）。

行为约定：
- 若某些日期无数据（非交易日/接口缺失）则不写入；
- 重跑回填不会产生重复记录（通过唯一键 + upsert 保证幂等）。

### 2) 每日增量同步（Daily Sync）

适用场景：
- 每天运行一次（例如北京时间固定时间）。

目标：
- 把“最新交易日”的收盘价与汇率同步进库；
- 顺带回补最近 N 天缺口以抵抗网络失败（建议 N=7）。

输入：
- 当前“已纳入采集”的标的集合；
- 同步窗口（例如最近 7 个自然日，实际落库按交易日）。

输出：
- 对每个标的：写入最新可用交易日收盘价；
- 写入最新可用日期的 USD/CNY 汇率。

行为约定：
- 如果当天未收盘（尤其是美股），以数据源返回的日期为准，不强行写“今天”；
- 若发现缺口（最近 N 天某日缺价格/汇率），尝试补拉一次；
- 同一天同标的数据允许覆盖更新（upsert）。

### 3) 新增标的（Add Symbol）

动作：
- 把标的加入“采集清单”（通常来自私有账本：持仓或交易里出现新代码）；
- 立即对该标的执行历史回填（Backfill）；
- 后续纳入每日增量同步（Daily Sync）。

## 日期与时区对齐（建议约定）

- **价格日期**：以数据源返回的“交易日日期字段”为准落库。
- **美股收盘时点**：北京时间早上运行时，美股可能尚未产生“当日收盘价”，因此同步到的会是“上一交易日”。
- **汇率日期**：优先按“价格日期”对齐使用；若汇率源按自然日提供，则以同日记录匹配价格日期（缺则允许回退到最近可用一日，但应在业务计算层明确规则）。

## 数据落库（建议的幂等与唯一性）

为保证“可重复运行/不会重复写”，建议以唯一键做 upsert：

- `stock_prices`：`(code, date, currency)` 唯一
- `exchange_rates`：`(from_currency, to_currency, date)` 唯一

并建议保留：
- `source`（数据来源标记）
- `created_at/updated_at`（用于排查与审计，非必须但很有用）

## 与其他层的边界

- 数据采集层只负责“把真实数据写进库”，不计算盈亏、不做成本口径。
- 业务计算层负责“缺口如何体现在日报里”（跳过、提示、或使用最近可用日等明确规则）。
- 交付层负责“生成并落地最新日报文件”，供你或 OpenClaw 读取转发。
