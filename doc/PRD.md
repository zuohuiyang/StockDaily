# StockDaily - 产品需求文档（PRD）

## 1. 背景与目标

### 1.1 背景

用户同时持有 A 股与美股，希望维护自己的持仓与成本信息，并在每天收盘后得到一份“可本地落盘”的投资组合日报。日报由 OpenClaw 读取并转发，因此本项目只需要稳定产出文件即可。

### 1.2 目标

- 提供一个本地可运行的流水线：私有账本（持仓/交易/成本） + 公网行情（收盘价/汇率） → 本地 Markdown 日报
- 覆盖 A 股与美股
- 日报落盘路径稳定、内容可读、缺失数据可追踪

### 1.3 成功标准（Success Metrics）

- 用户无需手工拷贝数据即可每日产出 `daily_reports/latest_report.md`
- 日报包含：估值日期、汇率、组合汇总、持仓明细、数据缺口提示
- 缺少行情或汇率时不会产生误导性的汇总值（缺失即缺失，显式提示）

## 2. 用户画像与使用场景

### 2.1 用户画像

- 个人投资者
- 使用 Windows 本地环境运行脚本
- 通过 OpenClaw 消费本项目产出的文件与结果

### 2.2 典型场景

1) 用户维护交易 CSV（或仅维护持仓）  
2) 每天运行一次 StockDaily（可手动或由外部调度触发）  
3) 产出 `daily_reports/latest_report.md`  
4) OpenClaw 读取该文件并进行转发/摘要/归档

## 3. 范围与非目标

### 3.1 范围（In Scope）

- A 股与美股的收盘价（EOD）采集与入库
- 至少 USD/CNY 汇率的入库与估值换算
- 基于“持仓/交易”计算组合估值
- 生成 Markdown 日报并落盘（latest + 归档）
- 数据缺失的显式提示与可观测性（来源、日期、缺口清单）

### 3.2 非目标（Out of Scope）

- 自动交易、策略回测、选股推荐
- 邮件/微信/飞书/Telegram 等主动推送（由 OpenClaw 或外部系统负责）
- 云端部署（GitHub Actions/Serverless）作为主运行形态
- 实时行情、盘中刷新（仅以收盘价为准）

## 4. 功能需求（FR）

### FR1：维护标的与持仓（P0）

用户需要告诉系统“要跟踪哪些标的、数量是多少、币种是什么”。

支持两种输入来源：

- 标准：symbols + transactions（推荐）
- 降级：holdings（仅估值，无成本盈亏）

验收要点：

- 在没有显式传入 symbols 列表时，系统能从数据库推导出活跃标的集合

### FR2：维护交易与成本（P0）

支持导入逐笔交易（BUY/SELL），用于计算：

- 成本均价（按成本口径）
- 已实现盈亏
- 未实现盈亏

数据项：

- trade_time, code, side, quantity, price, currency（必填）
- fee, fx_rate, broker, note（可选）

验收要点：

- 同一份 CSV 重复导入应具备幂等性（不应重复插入相同记录）

### FR3：每日采集（A 股/美股/汇率）并入库（P0）

每日采集内容：

- A 股：基于 6 位代码，采集日线收盘价（EOD）
- 美股：基于 ticker，采集日线收盘价（EOD）
- 汇率：至少 USD/CNY（用于换算 USD 仓位）

验收要点：

- 按日期写入 EOD 价格，不覆盖历史日期的其他币种主键
- 写入数据包含来源标识与抓取时间戳（用于追溯）

### FR4：生成日报并落盘（P0）

生成 Markdown 日报，包含：

- 报告时间（生成时刻）
- 估值日期（使用的 EOD 日期）
- USD/CNY（若缺失需显示“缺失”）
- 汇总：总资产（CNY）、未实现盈亏（CNY）、已实现盈亏（CNY）
- 明细表：标的、币种、数量、成本均价、收盘价、市值（CNY）、未实现盈亏（CNY）、已实现盈亏（CNY）
- 数据缺口：缺少收盘价/缺少汇率导致无法估值的标的列表

落盘规范：

- 归档：`daily_reports/YYYY-MM-DD_report.md`
- 最新：`daily_reports/latest_report.md`（稳定路径，供 OpenClaw 读取）

验收要点：

- 成功生成后 `latest_report.md` 必须存在且内容与当日归档一致

### FR5：缺失数据可观测（P0）

缺失定义：

- 标的在估值日期缺少收盘价
- USD 仓位在估值日期缺少 USD/CNY，导致无法换算 CNY

处理策略：

- 缺失标的在明细中保留行，但相关估值字段显示 `-`
- 汇总不纳入缺失项（避免产生误导）
- 在“数据缺口”章节列出缺失原因与标的清单

验收要点：

- 缺失情况下仍产出日报文件，且缺口信息可定位问题原因

## 5. 规则与口径（核心）

### 5.1 估值口径（EOD）

- 价格口径：收盘价（End Of Day）
- 估值日期：选取数据库中可用的最新 EOD 日期

### 5.2 成本与盈亏口径

- 默认成本口径：FIFO
- 未实现盈亏：以（收盘价 - 成本均价）× 持仓数量计算（在成本均价可得时）
- 已实现盈亏：以卖出成交额（扣费） - 对应卖出成本计算

说明：

- 若用户未导入交易，仅有 holdings，则成本均价与盈亏不可得，显示为 `-`

### 5.3 货币换算口径

- 第一阶段仅要求 USD→CNY
- 若 USD/CNY 缺失，则 USD 仓位不纳入 CNY 汇总，并在报告中提示缺失

### 5.4 缺失策略

- 缺失不插值、不估算
- 通过“数据缺口”明确提示，并给出用户可执行的回补建议（如回填历史、补齐汇率）

## 6. 输入输出与错误处理

### 6.1 输入

- 私有账本：交易 CSV 或 holdings 表
- 公网行情：A 股/美股收盘价、USD/CNY 汇率（由采集模块获取并入库）

### 6.2 输出

- Markdown 日报文件（latest + 归档）
- 进程退出码：
  - 0：成功生成日报
  - 非 0：关键错误（例如无法确定估值日期、数据库不可用）

### 6.3 错误处理原则

- 错误信息应可定位（缺少哪类数据、建议下一步操作）
- 不在日志/报告中输出密钥或敏感信息

## 7. 质量属性（NFR）

- 幂等：重复执行不会产生重复交易记录；价格/汇率按主键 upsert
- 可追溯：价格与汇率记录具备来源与抓取时间
- 稳定输出：latest 报告文件名与路径固定
- 安全：不在仓库中硬编码任何 API key

## 8. 验收标准（Acceptance Criteria）

### AC1：最小可用闭环（无交易）

前置条件：

- 数据库存在 holdings（或其他方式能推导出标的列表）
- 数据库存在目标日期的 prices_eod

验收：

- 生成 `daily_reports/latest_report.md`
- 报告包含估值日期、汇总、市值明细
- 成本均价与盈亏显示为 `-`（或 0 的口径须明确，不得误导）

### AC2：标准闭环（含交易 FIFO）

前置条件：

- 导入至少一笔 BUY 交易
- 存在目标日期 prices_eod 与 USD/CNY（若含 USD 仓位）

验收：

- 成本均价、已实现/未实现盈亏按 FIFO 口径可复算
- 汇总值等于明细可计入项之和

### AC3：缺失行情

前置条件：

- 标的存在，但估值日期缺少该标的 prices_eod

验收：

- 明细保留该标的行，收盘价/市值等为 `-`
- 汇总不包含该标的
- “数据缺口”列出该标的

### AC4：缺失汇率

前置条件：

- 存在 USD 仓位，但目标日期缺少 USD/CNY

验收：

- USD 仓位市值（CNY）为 `-`
- 汇总不包含 USD 仓位
- 报告显示 USD/CNY 为“缺失”并提示缺口

## 9. 后续迭代（不影响第一期验收）

- 港股与更多币种（HKD→CNY 等）
- 分红、拆股、税费口径
- 昨日/周度/月度对比维度与趋势图（仍以本地文件形式落盘）
