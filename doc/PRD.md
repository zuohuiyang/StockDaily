# StockDaily - 产品需求文档（PRD）

## 1. 背景与目标

### 1.1 背景

用户同时持有 A 股、美股与虚拟币（BTC、ETH），希望维护自己的持仓与成本价信息，并在每天收盘后得到一份“可本地落盘”的投资组合日报。日报由 OpenClaw 读取并转发：OpenClaw 运行在 Linux 上并消费本项目产出的固定目录下的日报文件，因此本项目只需要稳定产出文件即可。

### 1.2 目标

- 提供一个本地可运行的流水线：私有账本（持仓/成本价） + 公网行情（收盘价/汇率） → 本地 Markdown 日报
- 覆盖 A 股、美股与虚拟币（BTC、ETH）
- 日报落盘路径稳定、内容可读、缺失数据可追踪

### 1.3 成功标准（Success Metrics）

- 用户无需手工拷贝数据即可每日产出 `daily_reports/YYYY-MM-DD_report.md`
- 日报包含：报告日期、资产概览、持仓明细、数据缺口提示、对比信息（较昨日/较年初）
- 缺少行情或汇率时不会产生误导性的汇总值（缺失即缺失，显式提示）
- 私有账本输入为 JSON（禁止 CSV）

## 2. 用户画像与使用场景

### 2.1 用户画像

- OpenClaw 用户（个人投资者）
- StockDaily 在 Windows 本地环境运行并落盘
- OpenClaw 运行在 Linux 上，通过读取固定目录下的报告文件来消费结果（跨系统共享/同步方式不在本项目范围内）

### 2.2 典型场景

1) 用户维护本地持仓账本（非 CSV，包含数量与成本价）  
2) 每天运行一次 StockDaily（可手动或由外部调度触发）  
3) 产出 `daily_reports/YYYY-MM-DD_report.md`  
4) OpenClaw 读取该文件并进行转发/摘要/归档

## 3. 范围与非目标

### 3.1 范围（In Scope）

- A 股与美股的收盘价（EOD）采集与入库
- 虚拟币（BTC、ETH）的日线价格（EOD）采集与入库
- 至少 USD/CNY 汇率的入库与估值换算
- 基于“持仓（含成本价）”计算组合估值
- 生成 Markdown 日报并落盘（按日期归档）
- 数据缺失的显式提示与可观测性（来源、日期、缺口清单）

### 3.2 非目标（Out of Scope）

- 自动交易、策略回测、选股推荐
- 邮件/微信/飞书/Telegram 等主动推送（由 OpenClaw 或外部系统负责）
- 云端部署（GitHub Actions/Serverless）作为主运行形态
- 实时行情、盘中刷新（仅以收盘价为准）
- 第一阶段不做逐笔交易导入、FIFO 成本计算、已实现盈亏口径

## 4. 功能需求（FR）

### FR1：维护标的与持仓（P0）

用户需要告诉系统“要跟踪哪些标的、数量是多少、成本价是多少、币种是什么”。

持仓账本为唯一输入来源（第一阶段不做逐笔交易），最小字段集：

- asset_id：A 股 6 位代码 / 美股 ticker / 虚拟币 BTC、ETH
- asset_class：CN_STOCK / US_STOCK / CRYPTO
- quantity：数量
- currency：持仓币种（CNY / USD 等；虚拟币建议按 USD 计价）
- avg_cost：成本价（按持仓币种计价）

输入形态：JSON 文件（禁止 CSV）。

建议文件路径：`private/holdings.json`

示例：

```json
{
  "as_of": "2026-03-04",
  "positions": [
    { "asset_class": "CN_STOCK", "asset_id": "600519", "quantity": 10, "currency": "CNY", "avg_cost": 1500.0 },
    { "asset_class": "US_STOCK", "asset_id": "AAPL", "quantity": 5, "currency": "USD", "avg_cost": 160.0 },
    { "asset_class": "CRYPTO", "asset_id": "BTC", "quantity": 0.1, "currency": "USD", "avg_cost": 42000.0 }
  ]
}
```

字段约束：

- `as_of`：账本更新日期（用于排错，不作为报告日期）
- `positions`：持仓数组
- `quantity`：可为小数（适配虚拟币）

验收要点：

- 在没有显式传入 symbols 列表时，系统能从数据库推导出活跃标的集合

### FR2：每日采集（A 股/美股/虚拟币/汇率）并入库（P0）

每日采集内容：

- A 股：基于 6 位代码，采集日线收盘价（EOD）
- 美股：基于 ticker，采集日线收盘价（EOD）
- 虚拟币：仅 BTC、ETH，采集日线价格（EOD）
- 汇率：至少 USD/CNY（用于换算 USD 仓位）

验收要点：

- 按日期写入 EOD 价格，不覆盖历史日期的其他币种主键
- 写入数据包含来源标识与抓取时间戳（用于追溯）

缺失与可观测：

缺失定义：

- 标的在报告日期缺少收盘价
- 虚拟币（BTC、ETH）在报告日期缺少 EOD 价格
- USD 仓位在报告日期缺少 USD/CNY，导致无法换算 CNY

处理策略：

- 缺失不插值、不估算
- 缺失标的在报告明细中保留行，但相关估值字段显示 `-`
- 汇总不纳入缺失项（避免产生误导）
- 在报告的“数据缺口”章节列出缺失原因与标的清单

### FR3：生成日报并落盘（P0）

生成 Markdown 日报，包含：

- 报告日期（使用的 EOD 日期）
- 汇总：总资产（CNY）、昨日总资产（CNY）、较昨日（金额与百分比）、年初总资产（CNY）、较年初（金额与百分比）
- 明细表：标的、币种、数量、成本价、收盘价、市值（CNY）、较昨日（金额与百分比）、较年初（金额与百分比）
- 数据缺口：缺少收盘价/缺少汇率导致无法估值的标的列表

日报格式约定：

- 文件编码：UTF-8
- 数字展示：最多 2 位小数；缺失值展示为 `-`
- 行顺序：按 `asset_class`（CN_STOCK → US_STOCK → CRYPTO）再按 `asset_id` 升序

Markdown 模板（章节与标题固定）：

```md
# StockDaily 日报

- 报告日期：{{report_date}}

## 资产概览

| 指标 | 数值 |
|---|---:|
| 总资产（CNY） | {{total_value_cny}} |
| 昨日总资产（CNY） | {{total_value_prev_day_cny_or_missing}} |
| 较昨日（CNY） | {{delta_total_vs_prev_day_cny}}（{{delta_total_vs_prev_day_pct}}） |
| 年初总资产（CNY） | {{total_value_year_start_cny_or_missing}} |
| 较年初（CNY） | {{delta_total_vs_year_start_cny}}（{{delta_total_vs_year_start_pct}}） |

## 持仓明细

| 标的 | 类型 | 币种 | 数量 | 成本价 | 收盘价 | 市值（CNY） | 较昨日（CNY） | 较年初（CNY） |
|---|---|---|---:|---:|---:|---:|---:|---:|
| {{asset_id}} | {{asset_class}} | {{currency}} | {{quantity}} | {{avg_cost_or_missing}} | {{close_or_missing}} | {{value_cny_or_missing}} | {{delta_value_vs_prev_day_cny}}（{{delta_value_vs_prev_day_pct}}） | {{delta_value_vs_year_start_cny}}（{{delta_value_vs_year_start_pct}}） |

## 数据缺口

- 缺少收盘价：{{missing_prices_list_or_none}}
- 缺少汇率：{{missing_fx_list_or_none}}
```

## 5. 规则与口径（核心）

### 5.1 估值口径（EOD）

- 价格口径：收盘价（End Of Day）
- 报告日期：选取数据库中可用的最新 EOD 日期
- 虚拟币口径：BTC、ETH 的 EOD 以 UTC 日期为键；生成报告时按“报告日期”的同名日期取数

### 5.5 对比口径（昨日 / 年初）

- 较昨日：取小于报告日期的最近一个“可用报告日期”作为基准日
- 较年初：取当年 1 月 1 日之后（含）的第一个“可用报告日期”作为基准日
- 对比计算口径：使用“报告日期当天的持仓账本”，分别用报告日期与基准日的 EOD 价格/汇率进行估值，差额作为对比值
- 若基准日不存在，或基准日对应资产缺少可估值数据，则该对比值展示为 `-`

### 5.2 成本与盈亏口径

- 第一阶段成本口径：成本价由用户在持仓账本中维护（avg_cost）
- 第一阶段盈亏口径：不计算已实现/未实现盈亏，避免在缺少交易流水时产生误导

说明：

- 若持仓账本缺少成本价，则相关展示字段为 `-`

### 5.3 货币换算口径

- 第一阶段仅要求 USD→CNY
- 若 USD/CNY 缺失，则 USD 仓位不纳入 CNY 汇总，并在报告中提示缺失

### 5.4 缺失策略

- 缺失不插值、不估算
- 通过“数据缺口”明确提示，并给出用户可执行的回补建议（如回填历史、补齐汇率）

## 6. 输入输出与错误处理

### 6.1 输入

- 私有账本：持仓账本 JSON（包含数量与成本价）
- 公网行情：A 股/美股/虚拟币收盘价、USD/CNY 汇率（由采集模块获取并入库）

### 6.2 输出

- Markdown 日报文件（按日期归档）
- 进程退出码：
  - 0：成功生成日报
  - 非 0：关键错误（例如无法确定报告日期、数据库不可用）

### 6.3 错误处理原则

- 错误信息应可定位（缺少哪类数据、建议下一步操作）
- 当持仓账本缺少关键字段（如 quantity/avg_cost/currency）时，应明确指出缺失字段与对应标的，并避免输出误导性汇总
- 不在日志/报告中输出密钥或敏感信息

## 7. 质量属性（NFR）

- 幂等：持仓账本导入重复执行不会产生重复记录；价格/汇率按主键 upsert
- 可追溯：价格与汇率记录具备来源与抓取时间
- 稳定输出：归档目录与文件命名规则固定（按日期）
- 安全：不在仓库中硬编码任何 API key

## 8. 验收标准（Acceptance Criteria）

### AC1：最小可用闭环（持仓口径，第一期）

前置条件：

- 数据库存在持仓账本（或其他方式能推导出标的列表）
- 数据库存在报告日期的 prices_eod（覆盖 A 股/美股/虚拟币中至少一个资产类别）

验收：

- 生成 `daily_reports/YYYY-MM-DD_report.md`
- 报告包含报告日期、汇总、市值明细
- 报告包含“总资产较昨日/较年初”两行对比；若基准日或数据缺失则显示为 `-`
- 报告展示成本价；不展示或以 `-` 表示盈亏相关字段（不得误导）

### AC2：缺失行情（含虚拟币）

前置条件：

- 标的存在，但报告日期缺少该标的 prices_eod

验收：

- 明细保留该标的行，收盘价/市值等为 `-`
- 汇总不包含该标的
- “数据缺口”列出该标的

### AC3：缺失汇率

前置条件：

- 存在 USD 仓位，但报告日期缺少 USD/CNY

验收：

- USD 仓位市值（CNY）为 `-`
- 汇总不包含 USD 仓位
- 报告显示 USD/CNY 为“缺失”并提示缺口

## 9. 后续迭代（不影响第一期验收）

- 港股与更多币种（HKD→CNY 等）
- 分红、拆股、税费口径
- 昨日/周度/月度对比维度与趋势图（仍以本地文件形式落盘）
- 逐笔交易导入、FIFO 成本、已实现/未实现盈亏复算口径
